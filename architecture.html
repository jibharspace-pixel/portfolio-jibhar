Vue d'ensemble

Ce portfolio est une application full-stack composée de deux processus indépendants :

  • Frontend + Proxy Node.js (port 5000) — React 18 servi par Vite, Express en middleware.
  • API Rust / Axum (port 3001) — toutes les données, la logique métier et l'authentification admin.

Express proxifie les requêtes /api/* vers le serveur Rust. Les fichiers statiques sont servis par Vite en dev, ou par le build dist/ en production.

Mot de passe admin : nexalion2024 (ou variable ADMIN_PASSWORD côté Rust).
La validation est exclusivement côté serveur via POST /api/admin/verify — le mot de passe n'est jamais exposé dans le bundle JavaScript.
Frontend

React + TypeScript

React 18, Wouter, TanStack Query, Tailwind, shadcn/ui. Build via Vite.

Backend

Rust / Axum

API REST sur le port 3001. État en mémoire (Arc + Mutex). Multipart upload.

Proxy

Node.js / Express

Port 5000. Sert le frontend et proxifie /api/* vers Rust.

Footer dynamique

Le composant footer.tsx récupère les données via deux endpoints :

  • GET /api/contact → email, LinkedIn, WhatsApp, GitHub (liens dynamiques).
  • GET /api/site-content → champ footer_tagline (texte de présentation).

Tout le contenu du footer est modifiable depuis Admin → Informations sans redéploiement.

Animations de navigation

Deux systèmes en parallèle dans App.tsx :

  • View Transitions API (Chrome 111+, Safari 18+) — document.startViewTransition() avec animations CSS ::view-transition-old/new directionnelles.
  • Fallback CSS — classe page-enter-forward / page-enter-back sur la div key={location}.

Direction détectée via PAGE_ORDER : aller vers une page à droite dans la liste = forward, à gauche = back.

Conventions composants

  • Container : max-w-7xl mx-auto px-6 lg:px-8 sur toutes les sections.
  • ScrollReveal pour les animations d'entrée dans le viewport.
  • Constantes de catégorie dans lib/project-config.ts (icônes, labels, couleurs).
  • Tout fetch passe par TanStack Query — pas de useEffect + fetch directs.

Thème light/dark

Géré par ThemeProvider via localStorage clé portfolio-theme. Tailwind darkMode: ["class"]. La classe .dark est sur document.documentElement.

Backend Rust / Axum

Modules

models.rs

Types & Structs

Toutes les structs de données (Project, BlogPost, SiteContent…) et les types d'entrée (CreateProject, VerifyPayload…).

state.rs

AppState & Seed

AppState::new() initialise toutes les collections (Arc<Mutex<Vec<T>>>) et les données de démonstration.

handlers.rs

Handlers HTTP

Toutes les fonctions handler + helpers : check_admin_auth, verify_admin, projects_with_media, chrono_date.

main.rs

Entrée + Routeur

Déclare les modules, configure CORS, crée le Router Axum et démarre le serveur sur :3001.

Gestion de l'état

L'état est entièrement en mémoire (pas de base de données active). Chaque collection est un Arc<Mutex<Vec<T>>> partagé entre threads. Les données sont réinitialisées au redémarrage.

Pour persister les données, connectez PostgreSQL via DATABASE_URL et utilisez le schéma Drizzle dans shared/schema.ts.

Authentification admin

Deux mécanismes complémentaires :

  • Login frontend : POST /api/admin/verify — reçoit le mot de passe en JSON, répond 200 OK ou 401. Le mot de passe n'est jamais dans le bundle JS.
  • Appels API admin : chaque handler vérifie l'en-tête x-admin-password via check_admin_auth().

Upload de médias

Fichiers multipart/form-data → écrits dans rust_server/uploads/ avec un UUID comme nom. URL retournée : /uploads/<uuid>.ext. Le proxy Express sert ce dossier en statique.

Déploiement sur Replit

  1. Vérifier que les deux workflows sont actifs : Start application (port 5000) et Rust API Server (port 3001).
  2. Définir ADMIN_PASSWORD et ALLOWED_ORIGIN dans les Secrets Replit.
  3. Cliquer sur Deploy dans l'interface Replit — build et hosting automatiques.
Le dossier rust_server/uploads/ doit être persistant en production. Ne pas l'ajouter à .gitignore (ou monter un volume dédié sur un VPS).

Utiliser hors Replit

Prérequis

  • Node.js 18+
  • Rust stable + Cargo (rustup.rs)
  • Git

Lancer en développement local

  1. Cloner le projet : git clone <repo> && cd portfolio
  2. Installer les dépendances : npm install
  3. Démarrer Rust : cd rust_server && cargo run (terminal 1)
  4. Démarrer le frontend : npm run dev (terminal 2)
  5. Ouvrir http://localhost:5000

Variables à définir

ADMIN_PASSWORD=nexalion2024
ALLOWED_ORIGIN=http://localhost:5000
SESSION_SECRET=votre_cle_session

Build et démarrage en production

# Build frontend
npm run build

# Build Rust (release)
cd rust_server && cargo build --release && cd ..

# Lancer les deux processus
ALLOWED_ORIGIN=https://votredomaine.com \
ADMIN_PASSWORD=motdepassefort \
./rust_server/target/release/portfolio-api &

node ./dist/index.cjs
Maintenez ALLOWED_ORIGIN aligné avec l'URL publique du site pour éviter les erreurs CORS en production.

Ajouter une fonctionnalité

Nouvelle page frontend

  1. Créer client/src/pages/mapage.tsx avec export default function MaPage().
  2. Ajouter la route dans App.tsx : <Route path="/mapage" component={MaPage} />.
  3. Ajouter le lien dans components/navigation.tsx.
  4. Ajouter "/mapage" dans PAGE_ORDER dans App.tsx pour les transitions directionnelles.

Nouvel endpoint API Rust

  1. Ajouter la struct dans rust_server/src/models.rs.
  2. Ajouter le champ dans AppState et AppState::new() dans state.rs.
  3. Écrire le handler dans handlers.rs (pub async fn mon_handler(...)).
  4. Enregistrer la route dans main.rs (bloc Router::new()).

Ajouter une catégorie de projet

  1. Dans client/src/lib/project-config.ts : ajouter l'entrée dans CATEGORY_ICONS, CATEGORY_LABELS, CATEGORY_STYLES et PROJECT_FILTERS.
  2. Dans pages/admin/projects.tsx : ajouter l'option dans le <select> catégorie.
  3. Dans pages/admin/dashboard.tsx : ajouter la ligne dans la grille catégories.

Passer à PostgreSQL

  1. Définir DATABASE_URL dans les Secrets.
  2. Lancer npm run db:push pour appliquer le schéma Drizzle (shared/schema.ts).
  3. Remplacer les Arc<Mutex<>> dans state.rs par des appels SQL via sqlx dans les handlers.

Changer le mot de passe admin

  1. Définir la variable d'environnement ADMIN_PASSWORD=nouveaumdp côté Rust.
  2. Aucun changement à faire dans le code frontend — la validation est entièrement côté serveur.
