Vue d'ensemble

Ce portfolio est une application full-stack composée de deux processus indépendants :

  • Frontend + Proxy Node.js (port 5000) — React 18 servi par Vite, Express en middleware.
  • API Rust / Axum (port 3001) — toutes les données et la logique métier.

Express proxifie les requêtes /api/* vers le serveur Rust. Les fichiers statiques sont servis par Vite en dev, ou par le build dist/ en production.

Mot de passe admin : nexalion2024 (ou variable ADMIN_PASSWORD). En-tête HTTP : x-admin-password
Frontend

React + TypeScript

React 18, Wouter, TanStack Query, Tailwind, shadcn/ui. Build via Vite.

Backend

Rust / Axum

API REST sur le port 3001. État en mémoire (Arc + Mutex). Multipart upload.

Animations de navigation

Deux systèmes en parallèle dans App.tsx :

  • View Transitions API (Chrome 111+, Safari 18+) — document.startViewTransition() avec animations CSS ::view-transition-old/new directionnelles.
  • Fallback CSS — classe page-enter-forward / page-enter-back sur la div key={location}. Déclenche l'animation à chaque changement de route.

Direction détectée via PAGE_ORDER : aller vers une page à droite dans la liste = forward (slide de droite), aller vers une page à gauche = back (slide de gauche).

Conventions composants

  • Toutes les sections ont max-w-7xl mx-auto px-6 lg:px-8 pour le container.
  • ScrollReveal wrapping les éléments qui doivent s'animer à l'entrée dans le viewport.
  • Constantes visuelles de catégorie dans lib/project-config.ts (icônes, labels, couleurs).
  • Tout fetch passe par TanStack Query — pas de useEffect + fetch directs.

Thème light/dark

Géré par ThemeProvider via localStorage clé portfolio-theme. Tailwind darkMode: ["class"]. La classe .dark est sur document.documentElement.

Backend Rust / Axum

Modules

models.rs

Types & Structs

Toutes les structs de données (Project, BlogPost, etc.) et les types d'entrée (CreateProject, UpdateBlogPost…).

state.rs

AppState & Seed

AppState::new() initialise toutes les collections (Arc<Mutex<Vec<T>>>) et les données de démo.

handlers.rs

Handlers HTTP

Toutes les fonctions handler + helpers (check_admin_auth, projects_with_media, chrono_date).

main.rs

Entrée + Router

Déclare les modules, configure CORS, crée le Router Axum et démarre le serveur sur :3001.

Gestion de l'état

L'état est entièrement en mémoire (pas de base de données active). Chaque collection est un Arc<Mutex<Vec<T>>> partagé entre threads. Les données sont perdues au redémarrage.

Pour persister les données, connectez PostgreSQL via DATABASE_URL et utilisez le schéma Drizzle dans shared/schema.ts. Drizzle est déjà configuré — ajoutez les requêtes SQL dans handlers.rs via sqlx ou une couche de storage Node.js.

Authentification admin

Chaque handler admin appelle check_admin_auth(&headers) qui lit l'en-tête x-admin-password et le compare à la variable d'environnement ADMIN_PASSWORD (fallback : nexalion2024).

Upload de médias

Les fichiers sont uploadés via multipart/form-data, écrits dans rust_server/uploads/ avec un UUID comme nom. L'URL retournée est /uploads/<uuid>.ext. Le proxy Express sert ce dossier statique.

Déploiement

Sur Replit (recommandé)

  1. Vérifier que les deux workflows sont en marche : Start application (port 5000) et Rust API Server (port 3001).
  2. Définir ADMIN_PASSWORD et ALLOWED_ORIGIN dans les Secrets Replit.
  3. Cliquer sur Deploy dans l'interface Replit — le build et le hosting sont automatiques.

Sur un VPS / VM

# 1. Build frontend
npm run build

# 2. Build Rust (release)
cd rust_server && cargo build --release && cd ..

# 3. Lancer les deux processus
ALLOWED_ORIGIN=https://votredomaine.com \
ADMIN_PASSWORD=votre_secret \
./rust_server/target/release/portfolio-api &

node ./dist/index.cjs

Variables de production

ALLOWED_ORIGIN=https://kjs.portfolio.com
ADMIN_PASSWORD=motdepassefort123
DATABASE_URL=postgresql://user:pass@host/db
Le serveur Rust sert aussi les fichiers /uploads/* statiques. Assurez-vous que le dossier uploads/ est persistant (ne pas le gitignorer en prod).

Utiliser hors Replit

Prérequis

  • Node.js 18+
  • Rust stable + Cargo
  • Git

Lancer en local

  1. Cloner le projet.
  2. Installer les dépendances front : npm install.
  3. Démarrer le serveur Rust : cd rust_server && cargo run.
  4. Démarrer le frontend : npm run dev.
  5. Ouvrir http://localhost:5000.

cd rust_server
./target/release/portfolio-api

# Terminal 2
node ./dist/index.cjs
Si vous déployez sur un VPS ou un hébergement classique, gardez ALLOWED_ORIGIN aligné avec l’URL publique du site.

Ajouter une fonctionnalité

Nouvelle page frontend

  1. Créer client/src/pages/mapage.tsx avec export default function MaPage().
  2. Ajouter la route dans App.tsx : <Route path="/mapage" component={MaPage} />.
  3. Ajouter le lien dans components/navigation.tsx.
  4. Ajouter "/mapage" dans PAGE_ORDER dans App.tsx pour les transitions directionnelles.

Nouvel endpoint API

  1. Ajouter la struct dans rust_server/src/models.rs.
  2. Ajouter le champ dans AppState et AppState::new() dans state.rs.
  3. Écrire le handler dans handlers.rs (pub async fn).
  4. Enregistrer la route dans main.rs (bloc Router::new()).

Ajouter une catégorie de projet

  1. Dans client/src/lib/project-config.ts : ajouter l'entrée dans CATEGORY_ICONS, CATEGORY_LABELS, CATEGORY_STYLES et PROJECT_FILTERS.
  2. Dans rust_server/src/handlers.rs : mettre à jour le handler list_skills() si nécessaire.
  3. Dans pages/admin/projects.tsx : ajouter l'option dans le <select> catégorie.
  4. Dans pages/admin/dashboard.tsx : ajouter la ligne dans la grille catégories.

Passer à PostgreSQL

  1. Définir DATABASE_URL dans les Secrets.
  2. Lancer npm run db:push pour appliquer le schéma Drizzle (voir shared/schema.ts).
  3. Remplacer les Arc<Mutex<>> dans state.rs par des appels SQL via sqlx dans les handlers.
